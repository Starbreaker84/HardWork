## Избавляемся от зависимости от зависимостей-2



### Пример 1 

Зависимость: сервис, вызываемый контроллером в web-приложении:

- семантика: зависимости времени выполнения
- ключевые свойства: качество, безопасность, работоспособность
- пространство допустимых изменений: реализация сервиса может быть изменена без изменения API

### Пример 2

Зависимость: подключение Spring Security в web-приложение:

- семантика: зависимость сборки
- ключевые свойства: обеспечение безовасности при доступе к приложению и разграничение ролей при использовании, является частью экосистемы Spring
- пространство допустимых изменений: конфигурация Spring Security может быть подвержена изменениям, при этом контракт зависимости остаётся прежним

### Пример 3

Зависимость: применение системы логирования в web-приложении:

- семантика: функциональные требования
- ключевые свойства: возможность мониторинга исполнения запросов
- пространство допустимых изменений: система логированич может быть любоц, удовлетворяющей требованиям спецификации Spring Framework

### Пример 4

Зависимость: применение MongoDB в проекте:

- семантика: функциональные требования
- ключевые свойства: формат хранения данных
- пространство допустимых изменений: документированная СУБД может быть заменена на другую подобную при условии, что не будет использован специфический для вендора №№№функционал

### Выводы:
Расширенные факторы, определяющие зависимости, значительно улучшили мое понимание этого явления, особенно в контексте выявления реальных причин. Только обсуждая фактические проблемы и исключая гипотетические проблемы, такие как фреймврок, база данных и т.п., можно стремиться к четкости в рассуждениях.
